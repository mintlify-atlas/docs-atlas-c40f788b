## The Invariant

Faramesh enforces a fundamental ordering guarantee:

<Warning>
**WAL ORDERING INVARIANT**: Every governance decision has a durable audit record on disk **before** it is returned to the caller. No execution without an audit record — under any failure mode.
</Warning>

This means:
- If the decision is PERMIT, the agent executes **after** the record is fsynced
- If the WAL write fails, the decision is DENY (fail-closed)
- If the process crashes after WAL write but before execution, the WAL replays on restart
- No gap in the audit chain is possible

## The 11-Step Pipeline

Every CanonicalActionRequest flows through this pipeline:

```
CanonicalActionRequest
 │
 ├─[1]  Kill switch check      atomic.Bool per agent_id — nanoseconds
 ├─[2]  Phase check            tool visible in current workflow phase?
 ├─[3]  Pre-execution scanners shell, secret, PII — parallel goroutines
 ├─[4]  Session state read     sync.Map counters + ring buffer
 ├─[5]  History ring read      last N calls within T seconds
 ├─[6]  External selector fetch lazy, parallel, cached, timeout-bounded
 ├─[7]  Policy evaluation      expr-lang bytecode, first-match-wins, ~1μs/rule
 ├─[8]  Decision               PERMIT | DENY | DEFER | SHADOW
 ├─[9]  WAL write ──────────── fsync to local disk BEFORE returning decision
 ├─[10] Async                  replicate to SQLite DPR, update session + history
 └─[11] Return Decision        to adapter
```

**Step 9 is the critical one**: the WAL write happens **synchronously** before the decision is returned. Steps 10-11 are asynchronous.

## WAL Write Implementation

The WAL is implemented in `internal/core/dpr/wal.go`:

```go internal/core/dpr/wal.go:37
// Write appends a record to the WAL and calls fsync before returning.
// This must be called before the decision is delivered to the adapter.
func (w *WAL) Write(rec *Record) error {
    w.mu.Lock()
    defer w.mu.Unlock()

    line, err := json.Marshal(rec)
    if err != nil {
        return fmt.Errorf("marshal DPR record: %w", err)
    }
    line = append(line, '\n')

    if _, err := w.file.Write(line); err != nil {
        return fmt.Errorf("write WAL: %w", err)
    }
    if err := w.file.Sync(); err != nil {
        return fmt.Errorf("fsync WAL: %w", err)
    }
    return nil
}
```

Key properties:
- **Newline-delimited JSON**: Each record is a single line for fast parsing
- **fsync before return**: `w.file.Sync()` ensures the record is on disk
- **Mutex-protected**: Only one write at a time (serialized per agent)

<Info>
**fsync latency**: On modern SSDs, fsync typically takes 1-5ms. On HDDs, it can take 10-20ms. Use SSDs for production deployments.
</Info>

## Pipeline Integration

The pipeline calls the WAL in step 9:

```go internal/core/pipeline.go:348
// decide writes the WAL record and returns the Decision.
// This is the WAL ORDERING INVARIANT implementation:
// no decision is returned until the record is fsynced.
func (p *Pipeline) decide(req CanonicalActionRequest, d Decision, sess *session.State, start time.Time) Decision {
    d.Latency = time.Since(start)

    // Record metrics.
    observe.Default.RecordDecision(string(d.Effect), d.ReasonCode, d.Latency)

    // [9] WAL write — fsync before returning.
    rec := p.buildRecord(req, d)
    d.DPRRecordID = rec.RecordID
    if err := p.wal.Write(rec); err != nil {
        observe.Default.RecordWALWrite(false)
        // If we can't write the audit record, we must deny.
        // Execution must never precede the audit record.
        return Decision{
            Effect:     EffectDeny,
            ReasonCode: reasons.WALWriteFailure,
            Reason:     "audit record write failed; denying to preserve WAL invariant",
            Latency:    time.Since(start),
        }
    }
    observe.Default.RecordWALWrite(true)

    // [10] Async: replicate to SQLite, update session history, sync to Horizon.
    if p.store != nil {
        go func() {
            _ = p.store.Save(rec)
        }()
    }
    go sess.RecordHistory(req.ToolID, string(d.Effect))

    // [11] Return Decision.
    return d
}
```

<Warning>
**Critical**: If `p.wal.Write(rec)` returns an error, the decision is **immediately overridden to DENY**. The agent never sees a PERMIT without a durable record.
</Warning>

## Failure Modes

Let's analyze what happens under various failures:

### 1. Disk full

```
[9] WAL write → disk full → Write() returns error
    ↓
    Pipeline overrides decision to DENY
    ↓
[11] Return DENY to adapter
```

**Result**: Agent is denied. No execution. No missing audit record.

### 2. Process crash after WAL write, before return

```
[9] WAL write → fsync succeeds
    ↓
    Process crashes (OOM, SIGKILL, power loss)
    ↓
    Decision never returned to agent
    ↓
    Agent times out or retries
    ↓
    On restart: WAL replays, DPR chain is intact
```

**Result**: The audit record exists. The agent never received PERMIT, so it didn't execute. The DPR chain is complete.

### 3. Network partition (adapter can't reach core)

```
Adapter → (network partition) → Core
    ↓
    Adapter times out after 50ms
    ↓
    Adapter returns DENY to agent (fail-closed)
```

**Result**: Agent is denied. No execution. No audit record (because the core never saw the request).

<Note>
**Fail-closed guarantee**: If an adapter can't reach the core, the decision is DENY. Agents never execute without governance.
</Note>

### 4. SQLite write fails (async replication)

```
[9] WAL write → fsync succeeds
    ↓
[10] Async: SQLite write → fails (disk full, corruption)
    ↓
    Logged as error, metric incremented
    ↓
[11] Return Decision (already succeeded)
```

**Result**: The WAL is intact. SQLite is inconsistent. On restart, the WAL replays into SQLite to restore consistency.

## WAL Replay on Startup

When the daemon starts, it replays the WAL to ensure the SQLite store is consistent:

```go
func replayWAL(walPath string, store *dpr.Store) error {
    lines, err := readWALLines(walPath)
    if err != nil {
        return err
    }
    for _, line := range lines {
        var rec dpr.Record
        if err := json.Unmarshal(line, &rec); err != nil {
            return err
        }
        // Check if record already exists in SQLite
        exists, _ := store.Exists(rec.RecordID)
        if !exists {
            // Replay: insert into SQLite
            _ = store.Save(&rec)
        }
    }
    return nil
}
```

This ensures:
✅ Every WAL record eventually reaches SQLite  
✅ No duplicate records (checked by `Exists()`)  
✅ Crash recovery: SQLite is always consistent with WAL after startup

## Performance Characteristics

| Operation | Latency | Notes |
|-----------|---------|-------|
| Steps 1-8 (evaluation) | ~50μs | expr-lang bytecode, in-memory state |
| Step 9 (WAL write + fsync) | 1-5ms (SSD) | Dominates total latency |
| Step 10 (async SQLite) | 5-20ms | Doesn't block decision return |
| **Total latency** | **1-5ms** | Measured in production |

<Info>
**Optimization**: Faramesh batches WAL writes when processing multiple requests concurrently. This amortizes fsync cost across decisions.
</Info>

## Why Not Async WAL?

Why not write the WAL asynchronously and return the decision immediately?

**Because it would break the invariant**:

```
# WRONG: Async WAL
[8] Decision → PERMIT
[9] Async: WAL write (queued)
[11] Return PERMIT to adapter
    ↓
    Agent executes
    ↓
    Process crashes before WAL write completes
    ↓
    No audit record exists!
```

This is **unacceptable** for compliance and forensics. The WAL **must** be synchronous.

## Comparison with Other Systems

| System | Audit Guarantee | fsync? |
|--------|----------------|--------|
| **Faramesh** | No execution without audit record | Yes (before decision) |
| Traditional API Gateway | Audit after execution (best effort) | No |
| OPA (Open Policy Agent) | No built-in audit | N/A |
| AWS CloudTrail | Eventual consistency (minutes) | Async |
| Vault | Audit after execution | Yes (after execution) |

Faramesh is the only system that guarantees **audit before execution** with fsync.

## Observability

Monitor WAL health:

```bash
# WAL write latency
faramesh observe metrics | grep wal_write_latency_ms

# WAL write failures
faramesh observe metrics | grep wal_write_failures_total

# WAL replay on startup
faramesh observe metrics | grep wal_replay_duration_ms
```

<Warning>
If `wal_write_failures_total` increases, investigate disk health immediately. WAL failures result in DENY decisions.
</Warning>

## Best Practices

✅ **Use SSDs**: WAL fsync latency is critical path  
✅ **Monitor disk space**: WAL writes fail if disk is full  
✅ **Rotate WAL files**: Faramesh auto-rotates at 1GB by default  
✅ **Backup WAL**: WAL is the source of truth for audit  
❌ **Don't use NFS**: Network filesystems have unreliable fsync semantics

<Info>
For distributed deployments, replicate the WAL to S3/GCS after rotation for long-term retention.
</Info>
